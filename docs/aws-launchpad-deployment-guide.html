1. Requisitos Previos

Antes de iniciar el despliegue, asegurese de contar con los siguientes requisitos en su entorno de trabajo.

Cuenta AWS
Con acceso a Amazon Bedrock habilitado (Claude Sonnet 4.6)
Herramientas
Node.js 18+ y Docker instalados, o usar AWS CloudShell (ya los incluye)
AWS CDK
Bootstrapped en la cuenta y region destino
Permisos
Acceso de administrador en la cuenta AWS

Bootstrap de AWS CDK

Si es la primera vez que utiliza CDK en esta cuenta/region, ejecute el siguiente comando para preparar el entorno:

cdk bootstrap

Este comando crea un bucket S3 y los recursos necesarios para que CDK pueda desplegar stacks en su cuenta.

2. Despliegue Estandar (Cuenta Unitaria)

Para organizaciones que operan con una sola cuenta AWS. Este es el flujo mas comun y el punto de partida recomendado. El proceso completo toma aproximadamente 10 a 15 minutos.

1

Clonar el Repositorio

git clone https://github.com/aws-samples/aws-launchpad.git
2

Instalar Dependencias

cd aws-launchpad && npm install
3

Bootstrap de CDK (si no se hizo previamente)

cdk bootstrap
4

Desplegar el Stack

cdk deploy -c adminEmail=admin@empresa.com -c language=es
CDK despliega automaticamente los siguientes recursos: Amazon Cognito, API Gateway, CloudFront, DynamoDB, AgentCore Runtime, AgentCore Gateway, AgentCore Memory, funciones Lambda y Bedrock Guardrails.

3. Despliegue Multi-Cuenta (Payer + Linked Accounts)

Para partners o equipos que administran multiples cuentas bajo AWS Organizations. Este modo permite al agente consultar recursos en todas las cuentas vinculadas desde una unica interfaz.

1

Desplegar en la Cuenta Payer/Management

Ejecute el despliegue con el flag enableCrossAccount habilitado:

cdk deploy -c adminEmail=admin@empresa.com -c language=es -c enableCrossAccount=true
2

Configurar Acceso a Cuentas Linked

Una vez desplegado, el agente genera automaticamente un template de CloudFormation con un rol ReadOnly y los comandos necesarios para desplegarlo via StackSet (todas las cuentas) o individualmente en cada cuenta linked. Consulte la seccion 7 de esta guia para el detalle completo.

3

Verificar Acceso

Desde el chat del agente, solicite listar las cuentas de la organizacion. El agente confirmara cuales cuentas tienen el rol configurado y estan accesibles.

4. Parametros de Configuracion

Todos los parametros se pasan al comando cdk deploy usando la sintaxis -c parametro=valor. Solo adminEmail es obligatorio.

Parametro Requerido Default Descripcion
adminEmail Requerido - Email del administrador inicial (recibe contrasena temporal)
language Opcional en Idioma de la interfaz y el agente (en, es, pt)
modelId Opcional Claude Sonnet 4.6 ID del modelo de Bedrock o inference profile
domainName Opcional - Dominio personalizado (ej: launchpad.empresa.com)
hostedZoneId Opcional - ID de la Hosted Zone en Route 53
zoneName Opcional - Nombre de la zona en Route 53
enableCrossAccount Opcional false Habilitar visibilidad multi-cuenta

Ejemplo con Todas las Opciones

cdk deploy -c adminEmail=admin@empresa.com -c language=es -c modelId=us.anthropic.claude-sonnet-4-6 -c domainName=launchpad.empresa.com -c hostedZoneId=Z0123456789ABC -c enableCrossAccount=true

5. Primer Inicio de Sesion

Una vez completado el despliegue, CDK muestra los outputs del stack incluyendo la URL de CloudFront. Siga estos pasos para acceder por primera vez.

1

Obtener Credenciales

Revise el correo electronico configurado en adminEmail. Recibira un mensaje de Amazon Cognito con su contrasena temporal.

2

Acceder a la Aplicacion

Abra la URL de CloudFront que aparece en los outputs del CDK al finalizar el despliegue. Inicie sesion con su email y la contrasena temporal recibida.

3

Configurar Nueva Contrasena

El sistema le solicitara establecer una nueva contrasena segura en el primer inicio de sesion.

4

Configurar MFA

Configure la autenticacion multifactor (TOTP) usando una aplicacion compatible: Google Authenticator, Authy o Microsoft Authenticator. Este paso es obligatorio para todos los usuarios.

5

Comenzar a Usar el Asistente

Una vez autenticado, puede comenzar a interactuar con el asistente de operaciones en la nube directamente desde la interfaz de chat.

6. Gestion de Usuarios

LaunchPad utiliza Amazon Cognito para la gestion de identidades. El administrador tiene dos formas de agregar nuevos usuarios al sistema.

Desde la Interfaz
Haga clic en el icono de usuarios en el header de la aplicacion. Esto abre directamente la consola de Cognito User Pool donde puede crear y administrar usuarios.
Desde el Agente
Solicite al agente el comando CLI para crear usuarios. El agente generara el comando de AWS CLI listo para ejecutar.
Todos los usuarios tienen acceso exclusivamente de lectura. El agente nunca ejecuta acciones de escritura sobre los recursos de las cuentas AWS. Este modelo de seguridad garantiza que LaunchPad es seguro para uso en entornos de produccion.

7. Configuracion Multi-Cuenta (Detalle)

Esta seccion describe el flujo completo para habilitar el acceso a cuentas linked dentro de una organizacion de AWS. El proceso esta disenado para ser guiado por el propio agente.

Flujo de Configuracion

  • El partner abre LaunchPad y solicita acceso a las otras cuentas de la organizacion
  • El agente lista las cuentas disponibles en AWS Organizations
  • El agente genera un template de CloudFormation que crea el rol LaunchPadReadOnlyRole
  • El partner ejecuta el comando proporcionado en CloudShell o su terminal local
  • El agente verifica el acceso a cada cuenta configurada

Opcion A: StackSet (Todas las Cuentas)

Despliega el rol en todas las cuentas de la organizacion de forma automatica. Incluye auto-deployment para cuentas que se agreguen en el futuro.

aws cloudformation create-stack-set \ --stack-set-name LaunchPadAccess \ --template-body file://launchpad-role.yaml \ --capabilities CAPABILITY_NAMED_IAM \ --permission-model SERVICE_MANAGED \ --auto-deployment Enabled=true,RetainStacksOnAccountRemoval=false
aws cloudformation create-stack-instances \ --stack-set-name LaunchPadAccess \ --deployment-targets OrganizationalUnitIds=YOUR_OU_ID \ --regions us-east-1

Opcion B: Cuenta Individual

Para desplegar el rol en una cuenta especifica. Ejecute este comando desde la cuenta linked destino.

aws cloudformation deploy \ --template-file launchpad-role.yaml \ --stack-name LaunchPadAccess \ --capabilities CAPABILITY_NAMED_IAM

Seguridad del Rol Cross-Account

Aspecto Detalle
Nombre LaunchPadReadOnlyRole
Permisos ReadOnlyAccess (AWS managed policy)
Trust Solo el LaunchpadRuntimeRole de la cuenta payer
Escritura Ninguna - 100% read-only
Credenciales Temporales, 1 hora, renovacion automatica

8. Estimacion de Costos

El costo de LaunchPad depende principalmente del volumen de mensajes procesados por Amazon Bedrock. La siguiente tabla muestra estimaciones para tres niveles de uso tipicos.

Nivel Usuarios Mensajes/Mes Costo Estimado
Bajo 5 500 ~$7/mes
Medio 20 2,500 ~$35/mes
Alto 50 10,000 ~$140/mes
Los tokens de Amazon Bedrock representan aproximadamente el 85% del costo total. Amazon Cognito es gratuito hasta 10,000 usuarios activos mensuales (MAU). Los demas servicios (API Gateway, Lambda, DynamoDB, CloudFront) operan dentro del free tier para la mayoria de los casos de uso.

9. Eliminacion de Recursos

Para eliminar todos los recursos creados por LaunchPad, ejecute el siguiente comando desde el directorio del proyecto. DynamoDB y S3 se eliminan automaticamente gracias a la politica de remocion configurada en el stack (DESTROY).

Eliminar el Stack Completo

cdk destroy

Este comando elimina todos los recursos de AWS creados por el stack, incluyendo Cognito User Pool, API Gateway, CloudFront distribution, tablas DynamoDB, funciones Lambda y los roles IAM asociados.

Si desplegó roles cross-account via StackSet, elimine primero las instancias del StackSet y luego el StackSet antes de ejecutar cdk destroy.
