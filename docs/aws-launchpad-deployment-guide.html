1. Requisitos Previos

Antes de iniciar el despliegue, asegurese de contar con los siguientes requisitos en su entorno de trabajo.

Cuenta AWS
Cuenta activa (no se necesita habilitar modelos de Bedrock manualmente)
Entorno
AWS CloudShell (recomendado) o entorno local con Node.js 18+ y Docker
Permisos
Acceso de administrador en la cuenta AWS
No se necesita ejecutar cdk bootstrap manualmente. El script setup.sh lo ejecuta automáticamente como parte del proceso de instalación.

2. Despliegue Estandar (Cuenta Unitaria)

Para organizaciones que operan con una sola cuenta AWS. Este es el flujo mas comun y el punto de partida recomendado. El proceso completo toma aproximadamente 15 a 20 minutos.

1

Clonar el Repositorio

git clone https://github.com/RayihBou/AWS-Launch-Pad.git cd AWS-Launch-Pad
2

Ejecutar el Setup Interactivo

./setup.sh

El script interactivo te guiara a traves de la configuración:

  • Email del administrador (recibira credenciales temporales de Cognito)
  • Idioma de la interfaz (en/es/pt)
  • Visibilidad multi-cuenta (habilitar acceso a cuentas linked)

Proceso Automatizado

El script ejecuta 8 pasos de forma secuencial:

[1/8] Instalar dependencias del proyecto [2/8] Instalar dependencias del frontend [3/8] Compilar frontend (build inicial) [4/8] Bootstrap de CDK (configura la cuenta para despliegues) [5/8] Descargar imagen del agente (ECR Public → ECR privado) [6/8] Desplegar infraestructura (CDK deploy ~10-15 min) [7/8] Configurar frontend con endpoints del stack [8/8] Subir frontend a S3 + invalidar cache de CloudFront
Tiempo estimado: 15-20 minutos. El paso mas largo es el despliegue de infraestructura con CDK (~10-15 min). El email configurado como adminEmail recibira una contraseña temporal de Cognito al finalizar.

Despliegue Avanzado (CDK Directo)

Para usuarios que prefieran ejecutar los comandos de CDK manualmente sin el script interactivo:

npm install cd frontend && npm install && npm run build && cd .. npx cdk bootstrap aws://ACCOUNT_ID/REGION --app "" npx cdk deploy -c adminEmail=admin@example.com -c language=es -c containerUri=ACCOUNT.dkr.ecr.REGION.amazonaws.com/launchpad-agent:latest --require-approval never

Reemplace ACCOUNT_ID, REGION y ACCOUNT con los valores correspondientes a su cuenta AWS.

CDK despliega automáticamente los siguientes recursos: Amazon Cognito, API Gateway, CloudFront, DynamoDB, AgentCore Runtime, AgentCore Gateway, AgentCore Memory, funciones Lambda y Bedrock Guardrails.

3. Despliegue Multi-Cuenta (Payer + Linked Accounts)

Para partners o equipos que administran multiples cuentas bajo AWS Organizations. Este modo permite al agente consultar recursos en todas las cuentas vinculadas desde una unica interfaz.

1

Desplegar en la Cuenta Payer/Management

Si seleccionaste la opcion multi-cuenta durante el setup.sh, el despliegue ya incluye el flag enableCrossAccount=true. Si deseas habilitarlo despues de la instalación inicial:

npx cdk deploy -c adminEmail=admin@empresa.com -c language=es -c enableCrossAccount=true
2

Configurar Acceso a Cuentas Linked

El setup.sh genera automáticamente el archivo launchpad-role.yaml con el ARN real del Runtime role de tu cuenta. No es necesario pedirle al agente que genere el template. Al finalizar el despliegue, el script imprime los comandos necesarios para desplegar el rol via StackSet o individualmente.

3

Verificar Acceso

Desde el chat del agente, solicite listar las cuentas de la organización. El agente confirmara cuales cuentas tienen el rol configurado y estan accesibles.

4. Parámetros de Configuración

Todos los parámetros se pasan al comando cdk deploy usando la sintaxis -c parámetro=valor. Solo adminEmail es obligatorio.

Parámetro Requerido Default Descripcion
adminEmail Requerido - Email del administrador inicial (recibe contraseña temporal)
language Opcional en Idioma de la interfaz y el agente (en, es, pt)
modelId Opcional Claude Sonnet 4.6 ID del modelo de Bedrock o inference profile
domainName Opcional - Dominio personalizado (ej: launchpad.empresa.com)
hostedZoneId Opcional - ID de la Hosted Zone en Route 53
zoneName Opcional - Nombre de la zona en Route 53
enableCrossAccount Opcional false Habilitar visibilidad multi-cuenta
containerUri Opcional (auto-generado) URI de la imagen Docker del agente en ECR

Ejemplo con Todas las Opciones

cdk deploy -c adminEmail=admin@empresa.com -c language=es -c modelId=us.anthropic.claude-sonnet-4-6 -c domainName=launchpad.empresa.com -c hostedZoneId=Z0123456789ABC -c enableCrossAccount=true

5. Primer Inicio de Sesion

Una vez completado el despliegue, CDK muestra los outputs del stack incluyendo la URL de CloudFront. Siga estos pasos para acceder por primera vez.

1

Obtener Credenciales

Revise el correo electrónico configurado en adminEmail. Recibira un mensaje de Amazon Cognito con su contraseña temporal.

2

Acceder a la Aplicacion

Abra la URL de CloudFront que aparece en los outputs del CDK al finalizar el despliegue. Inicie sesion con su email y la contraseña temporal recibida.

3

Configurar Nueva Contraseña

El sistema le solicitara establecer una nueva contraseña segura en el primer inicio de sesion.

4

Configurar MFA

Configure la autenticación multifactor (TOTP) usando una aplicacion compatible: Google Authenticator, Authy o Microsoft Authenticator. Este paso es obligatorio para todos los usuarios.

5

Comenzar a Usar el Asistente

Una vez autenticado, puede comenzar a interactuar con el asistente de operaciónes en la nube directamente desde la interfaz de chat.

6. Gestión de Usuarios

LaunchPad utiliza Amazon Cognito para la gestión de identidades. El administrador tiene dos formas de agregar nuevos usuarios al sistema.

Desde la Interfaz
Haga clic en el icono de usuarios en el header de la aplicacion. Esto abre directamente la consola de Cognito User Pool donde puede crear y administrar usuarios.
Desde el Agente
Solicite al agente el comando CLI para crear usuarios. El agente generara el comando de AWS CLI listo para ejecutar.
Todos los usuarios tienen acceso exclusivamente de lectura. El agente nunca ejecuta acciones de escritura sobre los recursos de las cuentas AWS. Este modelo de seguridad garantiza que LaunchPad es seguro para uso en entornos de producción.

7. Configuración Multi-Cuenta (Detalle)

Esta sección describe el flujo completo para habilitar el acceso a cuentas linked dentro de una organización de AWS. El archivo launchpad-role.yaml se genera automáticamente durante el setup.sh con el ARN real del Runtime role.

Flujo de Configuración

  • El setup.sh genera el archivo launchpad-role.yaml con el ARN correcto del Runtime role
  • Al finalizar el despliegue, el script imprime los comandos de StackSet listos para ejecutar
  • El partner ejecuta los comandos en CloudShell o su terminal local
  • El agente verifica el acceso a cada cuenta configurada
Habilitar despues de la instalación inicial: Si no seleccionaste multi-cuenta durante el setup, puedes habilitarlo en cualquier momento ejecutando: npx cdk deploy -c enableCrossAccount=true con los demas parámetros de tu despliegue.

Opcion A: StackSet (Todas las Cuentas)

Despliega el rol en todas las cuentas de la organización de forma automática. Incluye auto-deployment para cuentas que se agreguen en el futuro. Estos comandos son impresos automáticamente por el setup.sh:

aws cloudformation create-stack-set \ --stack-set-name LaunchPadAccess \ --template-body file://launchpad-role.yaml \ --capabilities CAPABILITY_NAMED_IAM \ --permission-model SERVICE_MANAGED \ --auto-deployment Enabled=true,RetainStacksOnAccountRemoval=false
aws cloudformation create-stack-instances \ --stack-set-name LaunchPadAccess \ --deployment-targets OrganizationalUnitIds=YOUR_OU_ID \ --regions us-east-1

Opcion B: Cuenta Individual

Para desplegar el rol en una cuenta especifica. Ejecute este comando desde la cuenta linked destino.

aws cloudformation deploy \ --template-file launchpad-role.yaml \ --stack-name LaunchPadAccess \ --capabilities CAPABILITY_NAMED_IAM

Seguridad del Rol Cross-Account

Aspecto Detalle
Nombre LaunchPadReadOnlyRole
Permisos ReadOnlyAccess (AWS managed policy)
Trust Solo el LaunchpadRuntimeRole de la cuenta payer
Escritura Ninguna - 100% read-only
Credenciales Temporales, 1 hora, renovación automática

8. Estimación de Costos

El costo de LaunchPad depende principalmente del volumen de mensajes procesados por Amazon Bedrock. La siguiente tabla muestra estimaciónes para tres niveles de uso tipicos.

Nivel Usuarios Mensajes/Mes Costo Estimado
Bajo 5 500 ~$7/mes
Medio 20 2,500 ~$35/mes
Alto 50 10,000 ~$140/mes
Los tokens de Amazon Bedrock representan aproximadamente el 85% del costo total. Amazon Cognito es gratuito hasta 10,000 usuarios activos mensuales (MAU). Los demas servicios (API Gateway, Lambda, DynamoDB, CloudFront) operan dentro del free tier para la mayoria de los casos de uso.

9. Eliminación de Recursos

Para eliminar todos los recursos creados por LaunchPad, ejecute el siguiente comando desde el directorio del proyecto. DynamoDB y S3 se eliminan automáticamente gracias a la politica de remocion configurada en el stack (DESTROY).

Eliminar el Stack Completo

cdk destroy

Este comando elimina todos los recursos de AWS creados por el stack, incluyendo Cognito User Pool, API Gateway, CloudFront distribution, tablas DynamoDB, funciones Lambda y los roles IAM asociados.

Eliminar Repositorio ECR Privado

El setup.sh crea un repositorio ECR privado launchpad-agent que no se elimina con cdk destroy. Para eliminarlo manualmente:

aws ecr delete-repository --repository-name launchpad-agent --force --region us-east-1
Si desplegó roles cross-account via StackSet, elimine primero las instancias del StackSet y luego el StackSet antes de ejecutar cdk destroy.
